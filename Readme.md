本プロジェクトはPythonの機械学習ライブラリ(RandomForestRegressor)を用いて音楽ファイル(想定は.mp3)とEQを学習させたモデルを使い、新規曲に対してオススメEQを出力させるプロジェクトである。

本プロジェクトは情報解析を目的としており、著作権法第30条の4に基づき、特徴量のみを抽出・利用しています。音源そのものの配布は行っていません
また、実装予定の内容として外部(各デバイス)からファイルを受け取りそれを解析しオススメEQを渡すという機能を実装する予定だが、こちらもファイル管理はオンメモリ解析のみにし、サーバー上にファイルを残さない事で同法を回避するものとする。

制作経緯：
    著者は昔、音楽沼にドップリつかっていた時期があり、特に使用DAP(SONY NWa-55)でEQ設定を弄ることにはまった。が、ほぼ毎月追加する新曲のEQ設定をすることが次第に億劫になったり、そもDAP自身が記憶できるカスタマイズEQ設定が2個しかないため、あの曲の最適なEQで聞くと言ったことが難しくなっため次第に汎用的なEQのみの使用が増えていった。
    自分の好みのEQ設定の保存、設定していない曲に対してEQを提案を促してくれるものがあれば、と思い立ったため立ち上げたプロジェクトである。
    また、デプロイまで至った理由はもし外出先で保存したEQ、また新規曲のEQを知りたくなったとしてもスマホで確認できるようにするためである。


クラウドデータベース使用理由：デプロイ環境におけるエフェメラルなファイルシステム（再起動で消える仕組み）への対策として、外部DBへの移行が必要だと判断した結果そうした。また、DBのマイグレーション（設計変更）をアプリケーションのロジックから分離する(SQLiteでは毎回存在するか・しない場合は作成を確認していたがする必要が無くなるため)ことで、コードの保守性と堅牢性を高めるため。

データベースとしてSupabase(PostgreSQL)を使用：デプロイを考えた際に初期モジュールで導入の簡単なSQLiteではファイル形式なため同時に複数の対応が不可能かつ、より現場に近づけた環境テーブル形式で型を厳密に指定できるから。また集積・分析がFirebaseに比べ強く(そもFirebaseはNoSQLなため型を指定できない)、また機械学習に使いやすく、特徴量データをそのまま保存できる点などを加味した上でこれになった。(また基本的に無料で開発したいのでSupabaseを選んだ)
セキュリティはもしAPIキーがもれても本番環境では管理者権限のキーを使わないことで行の情報を抜き取らせないためにAnon KeyまたはPublishkey(Supabase上でRLSをONにした)

またデータベース設計では当たり前だが入力データにNOT NULLを適用し、NULLが返されエラーを吐かないかつ、空のカラムが無いようにした。
